# pydlock v2.0.0 (hardening) — Design

**Date:** 2026-07-08
**Status:** DRAFT — awaiting owner sign-off (plan §2 decisions)
**Plan:** [docs/plans/2026-07-08-repo-improvement-plan.md](../plans/2026-07-08-repo-improvement-plan.md)

## Context / problem

`pydlock` is a deliberately tiny tool: `python -m pydlock lock secret.txt` asks for a
password and encrypts a file in place; `unlock` reverses it. The library mirrors this with
`pydlock.lock(path)` / `pydlock.unlock(path)`. The **entire value proposition is that a
non-technical person can encrypt a file with one command**, so every change here is
constrained to preserve that: the mental model stays "enter a password."

Encryption uses `cryptography`'s **Fernet** (AES-128-CBC + HMAC-SHA256, authenticated,
timestamped) — a sound, vetted construction. The problem is **not** Fernet; it is the
key-derivation function feeding it. Verified in `pydlock/__init__.py` (2026-07-08):

```python
digest = sha256(password.encode(encoding)).hexdigest()   # fast, unsalted
key    = urlsafe_b64encode(digest.encode(encoding)[:32])  # first 32 hex chars → Fernet key
```

A **single, unsalted SHA-256** is wrong for password-based encryption on two axes:

1. **No salt.** Same password → same key, every time, for every user and every file.
   Identical plaintext produces correlatable ciphertext, and precomputed / rainbow-table
   attacks apply. There is no per-file uniqueness.
2. **No work factor.** SHA-256 is built to be fast; one round makes offline
   dictionary/brute-force against a captured file cheap. A password KDF must be deliberately
   slow and (ideally) memory-hard.

The repo is also **private on GitHub while the package is public on PyPI**, and is being
prepared to go public alongside two sibling repos this session. Going public surfaces two
more blockers and a batch of packaging debt:

- **Every commit carries the owner's private gmail** (`erickeshepherd@gmail.com`) — must be
  rewritten to the GitHub noreply identity before the repo is public.
- **Packaging debt:** `setup.py` declares **no `install_requires`** though the code imports
  `cryptography` (so `pip install pydlock` into a clean env breaks on first use); `__version__`
  renders with a **trailing dot** (`"1.3.0."`) and disagrees with PyPI's `1.2.0.15` and with
  `build.py`'s four-part scheme; no `pyproject.toml`, no tests, no CI.
- **Two latent correctness/security issues** found while reading: `encrypt`/`decrypt` use
  **text-mode I/O** and so corrupt binary files (the README's "Windows executable corruption"
  note is this bug, and it is fixable); and the `python`/`run` subcommands
  **decrypt-and-execute** arbitrary code (`exec` / `subprocess shell=True`).

(No secret keys or credentials exist anywhere in history — history is otherwise clean. The
`LICENSE` is already MIT; there is no relicense to do, unlike the sibling repos.)

## Goals & non-goals

**Goals**

- Replace the weak KDF with a **salted, tunable, memory-hard scrypt** KDF from `cryptography`,
  feeding the unchanged Fernet cipher.
- Store a small **versioned envelope** per encrypted file (`{version, kdf id, params, salt}`)
  so decryption stays "enter your password" and the format is future-proof.
- **Preserve trivial UX** — the one-line `lock`/`unlock` command and the `lock()`/`unlock()`
  API are unchanged in shape.
- **Strand no existing user:** transparently decrypt v1 files (in-tool legacy read); re-lock
  upgrades to v2.
- Fix binary-file corruption (bytes-mode I/O) and make writes atomic.
- Sanitize git history (private gmail → GitHub noreply) before the repo goes public.
- Modern packaging (`pyproject.toml`/hatchling, single-sourced `2.0.0`, declared
  `cryptography` dependency), full type hints, offline pytest suite, CI, and an OIDC
  Trusted-Publishing workflow. Publish-*ready*; publishing itself is owner-gated.

**Non-goals (scope fence)**

- **No custom crypto.** Keep Fernet; change only the KDF and the envelope around it.
- **No directory/archive encryption**, no obfuscate-compile "source protection" chains (the
  `FUTURE.rst` wishlist) — scope creep against the tiny-tool intent.
- **No key files, no asymmetric crypto, no keyrings.** Password-in, password-out stays the
  whole model.
- **No decrypt-and-execute features** — `python`/`run` are recommended for removal, not
  extension (see Key decisions / Open questions).
- **No publishing / no push / no making the repo public** inside the phases — all owner-gated.

## The envelope format (concrete)

A v1-locked file is a raw Fernet token — URL-safe base64 ASCII beginning `gAAAAA…` (Fernet
version byte `0x80`). v2 needs to store the salt and KDF params *with* the ciphertext while
staying self-identifying and binary-safe (plaintext is now bytes, and the envelope must never
be mistaken for a v1 token). Proposed layout — a fixed magic, a small JSON header, then the
Fernet token, newline-delimited:

```
b"PYDLOCK\x02\n"                          # magic + format version (0x02), unambiguous vs Fernet's gA…
<utf-8 JSON header><newline>              # {"kdf":"scrypt","n":32768,"r":8,"p":1,"salt":"<base64>"}
<Fernet token bytes>                      # AES-CBC+HMAC, exactly as today
```

- **Detection.** Starts with `b"PYDLOCK\x02"` → v2, parse the JSON header for salt+params.
  Otherwise → treat as **v1** and attempt the legacy scheme (below). The magic makes the two
  formats unambiguous.
- **Key derivation (v2).** `key = urlsafe_b64encode(Scrypt(salt, length=32, n, r, p).derive(password_bytes))`
  → a valid 32-byte Fernet key. Salt is 16 random bytes from `os.urandom`, fresh per lock.
- **Legacy read (v1).** `key = urlsafe_b64encode(sha256(password).hexdigest()[:32].encode())`,
  then `Fernet(key).decrypt(token)` — byte-for-byte the old derivation, used only on decrypt of
  a non-magic file.
- **Tunability / future-proofing.** `n/r/p` and the `kdf` id live in the header, so defaults
  can be raised over time and a `"pbkdf2"` id (with `iterations`) can be added without a new
  format version. Unknown `kdf` id → clear error, not a silent mis-decrypt.
- **Params start point:** scrypt `n=2**15 (32768), r=8, p=1` (interactive-login-grade,
  ~tens of ms, modest memory); revisit against the Phase-4 machines. PBKDF2 fallback:
  HMAC-SHA256, ~600k iterations (OWASP 2023).

JSON-header-over-binary-struct is chosen for legibility and painless extension at this scale;
the header is authenticated indirectly because a wrong salt/params yields a wrong key and
Fernet's HMAC then rejects the token (a tampered header cannot cause a silent wrong-plaintext,
only a clean decryption failure).

## Key decisions

- **Keep Fernet; fix only the KDF.** The cipher is sound and vetted; the defect is upstream of
  it. Minimal, auditable change — no custom crypto (plan non-goal, owner constraint).
- **scrypt with a per-file random salt** (owner-confirmable, plan D3). Memory-hardness raises
  brute-force cost beyond PBKDF2's; the `cryptography` `hazmat` `Scrypt` primitive is vetted.
  PBKDF2-HMAC-SHA256 is the documented fallback the envelope can carry for constrained
  environments.
- **Versioned envelope, not a bare token.** Storing `{version, kdf, params, salt}` is what lets
  decryption stay "enter your password" (the tool reads what it needs from the file) *and* lets
  the format evolve. Self-identifying magic makes v1/v2 detection unambiguous.
- **In-tool v1 legacy-decrypt** (owner-confirmable, plan D4). A breaking on-disk change must not
  strand data. `unlock` transparently reads v1 files with the old scheme; `lock` always writes
  v2, so re-locking upgrades a file. Keeps the one-line UX; no "downgrade pydlock to recover
  your files" dance. Alternative (documented `pydlock<2` procedure) is kept as a fallback in
  the README, not the primary path.
- **Bytes-mode I/O** (owner-confirmable, plan D6). Reading/writing bytes fixes binary-file and
  Windows-executable corruption (the README's documented "unfixable" bug — it *is* fixable) and
  removes the file-content `encoding` coupling. Strictly better; part of the major bump.
- **Atomic writes** (temp file + `os.replace`). In-place `"w+"` truncation risks data loss on
  interruption; atomic replace makes lock/unlock crash-safe while preserving in-place UX.
- **Remove `python` and `run`** (owner-confirmable, plan D5 — the main judgment call).
  Decrypt-and-execute (`exec`, `shell=True`) is a security footgun and scope creep vs. "encrypt
  a file for a non-technical user"; `run` can also leave the file decrypted on disk if
  interrupted mid-cycle. Removing them shrinks the CLI to the two verbs that are the product.
- **Keep the name `pydlock`; keep MIT** (plan D1/D2). Already public on PyPI under this name;
  MIT `LICENSE` already present. Metadata hygiene only (SPDX field, single-source version,
  declared dependency).
- **Fixtures committed; tests never touch the network** (there is no network anyway) — the
  suite is pure local crypto round-trips including a committed **v1 fixture** (file + known
  password) so legacy support can't silently regress.

## Alternatives considered

- **PBKDF2-HMAC-SHA256 as the primary KDF** — perfectly acceptable and slightly simpler to
  reason about, but not memory-hard; scrypt is the stronger default and PBKDF2 stays available
  as a carried fallback id. (Argon2id would be stronger still but needs `argon2-cffi`, an extra
  dependency beyond `cryptography` — deferred; the envelope can add an `"argon2id"` id later
  without a format break.)
- **Store salt as a fixed-width binary prefix instead of a JSON header** — marginally smaller,
  but less legible and harder to extend (adding PBKDF2's `iterations` or Argon2's `m/t/p` means
  a new binary layout). JSON header costs a few dozen bytes and buys painless evolution.
- **No legacy read — hard cut, documented downgrade path only** — simplest code, but strands
  users mid-upgrade and breaks the trivial-UX promise ("why won't my file open?"). Rejected in
  favor of transparent v1 read (D4).
- **Keep text-mode I/O, just document the binary limitation** — perpetuates a real corruption
  bug that already bites (Windows executables) for no benefit. Rejected.
- **Keep `python`/`run` behind a warning** — the fallback if the owner wants them (D5); still
  carries the execute-arbitrary-code surface. Recommended against.
- **Auto-re-lock v1 files to v2 on unlock** (silently upgrade in place) — surprising for a tool
  whose contract is "unlock gives me my file back"; a user expecting the file unchanged after
  `unlock` would find its on-disk format changed. Rejected in favor of explicit re-`lock`.
- **Rename / relicense** — both moot here (published name, already MIT). Rejected.

## Risks

- **History rewrite / force-push mistakes.** Mitigation: full backup clone before Phase 1;
  email-only rewrite is mechanical and diff-verifiable (`git log --format='%ae %ce'`);
  force-push is a deliberate, owner-run tail step.
- **A v1 file fails to decrypt after upgrade.** Mitigation: in-tool legacy path + a committed
  v1 test fixture (file + known password) that pins byte-for-byte compatibility; plus the
  documented `pydlock<2` fallback.
- **scrypt too memory-hungry / unavailable somewhere.** Mitigation: params are per-file and
  tunable; PBKDF2 is a carried fallback id; defaults are validated on the Phase-4 CI matrix.
- **Breaking CLI change (dropping `python`/`run`) surprises a user.** Low (niche features);
  CHANGELOG calls it out, and D5 is owner-confirmable.
- **Envelope/token tampering.** Non-issue for confidentiality: a wrong salt/params or a mangled
  token yields a wrong key or fails Fernet's HMAC → clean decryption failure, never a silent
  wrong-plaintext. (pydlock protects data at rest; it does not defend against an attacker who
  can rewrite the ciphertext file, and never claimed to.)

## Rollout / migration

- **Sequencing:** Phase 1 (history hygiene) first as a supervised rewrite; Phases 2→5 as
  branch → `pre-merge-review` → merge. Stop-anywhere is safe after Phase 3 (files encrypt with
  the hardened KDF).
- **Versioning:** everything ships as **v2.0.0** (breaking on-disk format, breaking CLI if
  `python`/`run` are removed, bytes-mode I/O). This resets the malformed `1.2.0.15` /
  `"1.3.0."` versioning to a clean single-sourced `2.0.0`.
- **User migration:** v1 files decrypt transparently; re-`lock` upgrades them to the salted v2
  envelope. Documented in the README and CHANGELOG; a `pydlock<2` procedure is the fallback.
- **Owner-gated tail (outside all phases):** (1) force-push the rewritten history to the
  private remote; (2) make the GitHub repo public; (3) cut the `v2.0.0` GitHub release →
  gated `publish.yml` → PyPI (republish over the existing project). Checklist is a Phase-5
  deliverable.
